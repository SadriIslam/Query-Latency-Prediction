## Project Summary

This project builds a hospital database using synthetic EHR (Electronic Health Record) data from [Synthea](https://synthea.mitre.org/), runs benchmark queries against it using PostgreSQL's `EXPLAIN ANALYZE` tool, and trains supervised ML regression models to predict query execution latency.

The central research question:

> **Can EML < EPG?**
> (Can ML model error be lower than PostgreSQL planner error?)

**Result:** Yes — Gradient Boosting achieved MAE of **54.4 ms**, which is **18% better** than PostgreSQL's baseline of **66.4 ms**. Random Forest also outperformed at **59.9 ms**.

## Dataset

The dataset `benchmark_dataset_60.csv` contains 60 benchmark queries with 6 features extracted from PostgreSQL `EXPLAIN ANALYZE` output.

### Features (inputs)

| Feature | Description |
|---------|-------------|
| `est_cost` | PostgreSQL's internal estimated cost (dimensionless) |
| `est_rows` | Estimated number of rows to be processed |
| `num_joins` | Number of table joins in the query |
| `join_type` | Join strategy: Hash=1, Nested Loop=2, Merge=3 |
| `scan_type` | Scan method: Sequential=1, Index=2, Bitmap=3 |
| `planning_time` | Time PostgreSQL spent planning (ms) |

### Target (output)

| Column | Description |
|--------|-------------|
| `execution_time` | Actual query execution time in milliseconds |

### Dataset Statistics

| Metric | Value |
|--------|-------|
| Total queries | 60 |
| Training queries | 48 (80%) |
| Test queries | 12 (20%) |
| Min execution time | 0.102 ms |
| Max execution time | 1073 ms |
| Mean execution time | ~119 ms |

### Query Type Breakdown

| Category | Count | Latency Range |
|----------|-------|---------------|
| Fast indexed joins | 27 | < 5 ms |
| Medium queries | 18 | 5 – 100 ms |
| Slow hash joins | 11 | 100 – 500 ms |
| Very slow full scans | 4 | 500 ms+ |

## Key Finding

The most important result from the feature importance analysis:

> **`est_cost` accounts for 68.2% of predictive power.**

This means the ML model is not replacing PostgreSQL's cost estimator — it is learning to *calibrate and correct* it. The model takes the planner's estimate as its primary input and adjusts for cases where the cost-to-time relationship is non-linear.

`planning_time` was the second most important feature at **20.8%**, confirming that the effort PostgreSQL expends in planning a query is itself a useful signal about query complexity.
